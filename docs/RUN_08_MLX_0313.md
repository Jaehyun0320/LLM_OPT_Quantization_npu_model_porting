# MLX-LM 0.31.3 Gemma 4 Retry

This project now has a dedicated MLX virtual environment at `.venv-mlx`.

## Environment

```bash
source .venv-mlx/bin/activate
python --version
python -m pip show mlx-lm mlx
```

Expected versions:

```text
Python 3.12.13
mlx-lm 0.31.3
mlx 0.31.2
```

This environment includes `mlx_lm.models.gemma4` and `mlx_lm.models.gemma4_text`, unlike the previous system Python environment with `mlx-lm 0.29.1`.

## Gemma 4 E2B Conversion Retry

Run this from a normal macOS terminal, not from a headless/sandboxed session, so that MLX can access the Metal device.

```bash
.venv-mlx/bin/python scripts/08_convert_edge.py \
  --model-id google/gemma-4-E2B \
  --mlx-path models/gemma4_e2b_mlx_4bit_0313 \
  --prompt "Deep learning is" \
  --max-tokens 32 \
  --output results/edge_mlx_conversion_0313.json
```

If memory is tight on an 8GB MacBook, first test conversion only:

```bash
.venv-mlx/bin/python scripts/08_convert_edge.py \
  --model-id google/gemma-4-E2B \
  --mlx-path models/gemma4_e2b_mlx_4bit_0313 \
  --skip-generate \
  --output results/edge_mlx_conversion_0313_convert_only.json
```

## Notes

- The old failure in `results/edge_mlx_conversion.json` came from `mlx-lm 0.29.1`, which did not include a local Gemma 4 backend.
- The new environment has the Gemma 4 backend, so the previous `No module named mlx_lm.models.gemma4` error should be resolved.
- Conversion may still fail if the Mac runs out of unified memory or if MLX cannot access Metal.
- In the Codex sandbox, MLX reported `No Metal device available`; run the conversion from your regular Terminal app.
